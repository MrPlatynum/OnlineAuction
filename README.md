## Описание системы

Лотус - это веб-приложение для проведения электронных торгов: пользователи выставляют лоты, делают ставки и следят за ходом аукциона в реальном времени. Лот закрывается автоматически по таймеру, средства победителя переводятся продавцу за вычетом комиссии платформы, а все участники получают уведомления в интерфейсе, по WebSocket и на почту.

Приложение построено как асинхронный монолит на FastAPI со слоистой архитектурой и отдельным слоем координации (планировщик завершения лотов, выбор лидера, outbox-воркер рассылки писем, менеджер WebSocket-соединений).

### Возможности

- **Аукционы** - создание лотов с фотографиями, категориями и стартовой ценой; режимы «торги» и «купить сейчас».
- **Ставки в реальном времени** - цена и таймер обновляются у всех зрителей лота через WebSocket без перезагрузки страницы.
- **Защита от снайпинга** - ставка в последние 120 секунд автоматически продлевает время лота (с ограничением до 30 продлений против сговора).
- **Баланс и расчёты** - депозит/вывод средств, учёт «замороженной» суммы у лидера торгов, атомарный перевод победитель -> продавец с аудит-журналом транзакций и комиссией платформы 7%.
- **Уведомления** - перебитая ставка, новая ставка по вашему лоту, скорое завершение, победа/проигрыш, продажа. Доставка по трём каналам: в приложении, по WebSocket и на email.
- **Профили и репутация** - публичные профили продавцов, отзывы с оценкой 1-5, подписки на продавцов и уведомления о новых лотах.
- **Загрузка изображений** - аватары и фото лотов с валидацией и санитизацией через Pillow, асинхронной записью на диск.
- **Безопасность** - JWT-аутентификация, хеширование паролей Argon2id, ограничение частоты запросов, блокировка аккаунта после серии неудачных входов, массовый отзыв сессий через `token_version`.

## Архитектура

Приложение разделено на прикладные слои и слой координации фоновых задач.

```
app/
  routers/    HTTP- и WebSocket-эндпоинты (49 REST + 2 WS)
  services/   бизнес-логика (аукционы, баланс, уведомления, email, планировщик)
  models/     SQLAlchemy-модели (10 таблиц)
  schemas/    Pydantic-схемы валидации
  utils/      безопасность, работа с деньгами, общие помощники
```

Ключевые асинхронные решения:

- **Завершение лотов через `asyncio.Task`** - каждый активный лот обслуживается самоперепланирующимся таймером. При старте приложение восстанавливает таймеры по активным лотам.
- **Выбор лидера через `pg_try_advisory_lock`** - при нескольких процессах фоновые задачи выполняет только один экземпляр; координатором выступает сам PostgreSQL, без ZooKeeper/Redis.
- **Outbox-паттерн для email** - письма сначала складываются в таблицу в одной транзакции с бизнес-операцией, затем воркер рассылает их с `SELECT FOR UPDATE SKIP LOCKED`, экспоненциальными повторами и dead-letter после исчерпания попыток. Гарантия доставки «как минимум один раз».

### Модель данных

Схема состоит из 10 таблиц. Денежные поля хранятся в `Numeric(12, 2)`, временные метки - с часовым поясом.

| Таблица | Назначение |
|---------|------------|
| `users` | Учётные записи, баланс, флаги верификации и настройки уведомлений. |
| `auctions` | Лоты: цены, сроки, тип (`bid`/`bin`), счётчик продлений, продавец и победитель. |
| `bids` | Ставки по лотам с суммой и временем (поле `id` - тай-брейкер при одинаковом времени ставок). |
| `auction_images` | Галерея изображений лота (удаляется каскадно вместе с лотом). |
| `categories` | Дерево категорий с самоссылкой `parent_id`. |
| `transactions` | Аудит-журнал движения средств (депозит, вывод, перевод, комиссия). |
| `notifications` | Уведомления в приложении с признаком прочтения. |
| `reviews` | Отзывы покупателей о продавцах с оценкой 1-5. |
| `subscriptions` | Подписки пользователей на продавцов. |
| `email_outbox` | Очередь исходящих писем (outbox-паттерн) со статусом и счётчиком попыток. |

## Как это работает

### Жизненный цикл аукциона

1. **Создание лота** (`POST /api/auctions`). Продавец задаёт стартовую цену, длительность и тип (`bid` - торги или `bin` - купить сейчас). Лот становится активным, а на его время завершения планируется фоновая задача.
2. **Ставка** (`POST /api/bids`). Строки пользователей блокируются (`SELECT ... FOR UPDATE`), проверяется эффективно доступный баланс (баланс за вычетом сумм, замороженных на других лотах, где пользователь лидирует), обновляется текущая цена. Прежнему лидеру уходит уведомление «перебито», всем зрителям лота - обновление по WebSocket.
3. **Анти-снайпинг.** Ставка в последние 120 секунд продлевает время лота и переносит задачу завершения - до 30 раз, чтобы исключить бесконечное затягивание.
4. **Завершение** (`complete_auction`). По таймеру лидер-процесс закрывает лот: определяет победителя (автор последней, наивысшей принятой ставки; при одинаковом времени - ставка с большим `Bid.id`), переводит средства продавцу за вычетом комиссии 7%, пишет транзакции и рассылает уведомления (победа/проигрыш/продажа) по трём каналам.
5. **Купить сейчас** (`settle_bin_purchase`). Для лота типа `bin` покупка по фиксированной цене мгновенно закрывает лот и проводит расчёты.

### Надёжность фоновых задач

- При нескольких процессах планировщик и outbox-воркер выполняет только лидер, захвативший `pg_try_advisory_lock`; остальные процессы ждут и перехватывают роль, если лидер останавливается.
- Письма записываются в таблицу `email_outbox` в одной транзакции с бизнес-операцией. Воркер выбирает их `FOR UPDATE SKIP LOCKED`, отправляет по SMTP с экспоненциальной задержкой между повторами и переводит в dead-letter после исчерпания попыток - доставка «как минимум один раз».

### Реальное время

- Менеджер WebSocket-соединений хранит подключения по лотам и пользователям и рассылает события (новая ставка, продление, завершение, уведомление) соответствующим подписчикам.

## Запуск

### Требования

- Python 3.13
- Docker и Docker Compose (для PostgreSQL)

### 1. Поднять базу данных

```bash
docker compose up -d postgres
```

Контейнер слушает `127.0.0.1:5433` и при первом запуске создаёт основную БД `auction` и тестовую `auction_test`.

### 2. Установить зависимости

Создайте и активируйте виртуальное окружение.

Linux / macOS:

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Windows (PowerShell):

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

### 3. Настроить окружение

Скопируйте шаблон в `.env`.

Linux / macOS:

```bash
cp .env.example .env
```

Windows (PowerShell):

```powershell
Copy-Item .env.example .env
```

Сгенерируйте секретный ключ и впишите его в `.env`:

```bash
python -c "import secrets; print(secrets.token_urlsafe(64))"
```

Значения по умолчанию в `.env.example` уже согласованы с `docker-compose.yml`. Минимально обязательная переменная - `AUCTION_SECRET_KEY`. SMTP-поля можно оставить пустыми: тогда письма не отправляются, но остальные каналы уведомлений работают.

### 4. Применить миграции

```bash
alembic upgrade head
```

### 5. Запустить приложение

```bash
uvicorn main:app --reload --port 8000
```

- Приложение: http://localhost:8000
- Документация API (Swagger UI): http://localhost:8000/docs

Категории создаются автоматически при первом запуске.

### Демо-данные (опционально)

Скрипт наполняет базу демо-пользователем и 25 активными лотами (категории должны быть засеяны - они создаются при первом запуске приложения). Защита от запуска на проде требует явной переменной `AUCTION_ENV`:

```bash
AUCTION_ENV=dev python -m scripts.seed_demo_auctions
```

```powershell
$env:AUCTION_ENV="dev"; python -m scripts.seed_demo_auctions
```

Пароль созданного демо-пользователя печатается один раз в конце работы скрипта.

### Конфигурация

Переменные окружения (см. `.env.example`):

| Переменная | Назначение |
|------------|------------|
| `AUCTION_SECRET_KEY` | Обязательно. Ключ подписи JWT (HS256). |
| `LEGACY_PASSWORD_KEYS` | Опционально. Ключи для проверки устаревших хешей паролей. |
| `DATABASE_URL` | Строка подключения к PostgreSQL (драйвер asyncpg). |
| `SMTP_SERVER` / `SMTP_PORT` / `SMTP_USERNAME` / `SMTP_PASSWORD` | Параметры SMTP для отправки писем. |
| `EMAIL_FROM` | Адрес отправителя в письмах. |
| `CORS_ORIGINS` | Список разрешённых источников (через запятую). |
| `AUCTION_RATE_LIMIT_ENABLED` | Включение ограничения частоты запросов (по умолчанию `true`). |

## Тестирование

Проект покрыт **291 тестом** (25 файлов), общее покрытие - **91%**. Используется pytest с асинхронным режимом; тесты работают с отдельной БД `auction_test`.

```bash
pip install -r requirements-dev.txt

# Все тесты
pytest

# С отчётом о покрытии
pytest --cov=app --cov-report=term-missing
```

Линтер:

```bash
ruff check .
```

> Не запускайте несколько процессов pytest одновременно - они используют общую тестовую БД и будут мешать друг другу при пересоздании схемы.
